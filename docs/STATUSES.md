# Справка по статусам дислокации

Статус вагона — производная величина, которую движок вычисляет на **Stage 1**
(обогащение) для каждой записи снимка. Он отражает, **где вагон и что с ним
происходит** (на отправлении / в пути / простаивает / брошен / прибыл / выгружен).
На статус опираются последующие шаги: прогноз хода, кандидаты прибытия, доноры
перегруза, простановка плана подвода.

- **Где считается:** `internal/service/enrich.go` → `computeStatus()` (§3.13).
- **Когда:** Stage 1, шаг 4 «статусы и производные» (после станций, порта, операций).
- **Тип:** `dislocation.status` (`*int`), распределение по батчу — в `StatusDist`.

> Время везде московское naive (без таймзоны). «Сейчас» — из `clock.Now()`.

---

## Дерево расчёта (порядок важен)

Статус определяется **первым** подходящим условием сверху вниз. Ключевые признаки:

- **`atDest`** — вагон на станции назначения: `station_oper == stan_nazn` (обе непусты).
- **`ПорожПризнак`** (`porozh_priznak == "1"`) — вагон порожний.

```
1. Порожний? ──да──► atDest? ──да──► 12  (порожний в порту / выгружен)
   │                 └──нет─► 6   (порожний в пути → донор перегруза)
   │нет
2. atDest (гружёный на назначении)? ──да──► date_prib заполнен? ──да──► 10 (прибыл)
   │                                        └──нет──────────────► 9  (кандидат в прибывшие)
   │нет
3. На станции отправления (code_station_nach == code_station_oper)?
   │            ──да──► Index == "Б/И"? ──да──► 0 (без индекса)
   │                                     └нет─► 1 (на отправлении)
   │нет
4. Брошен (code_oper == "92")? ──да──► 5
   │нет
5. Долгий простой в пути (простой ≥ порога)? ──да──► 4
   │нет
6. ────────────────────────────────────────────────► 2 (в пути)
```

## Каждый статус подробно

### 0 — на станции отправления, без индекса
Вагон стоит на своей станции отправления, но поездного индекса ещё нет
(`Index == "Б/И"`). Операционный буфер: считается «ещё не в ходу», поэтому
**прогноз прибытия для него не строится** (`forecast.go:85`).

### 1 — на станции отправления
То же место (ст. отправления = ст. операции), но индекс уже присвоен — вагон
готов к отправлению / формируется. Прогноз хода строится.

### 2 — в пути
Значение по умолчанию: ни одно спец-условие не сработало. Вагон движется между
станцией отправления и назначения. Основной «рабочий» статус для расчёта хода.

### 4 — долгий простой в пути
Вагон стоит в пути (станция операции ≠ отправления и ≠ назначения) дольше порога:
`prost_dn ≥ prost_dn_min` (сутки) **или** `prost_ch ≥ prost_ch_min` (часы).
Пороги — из настроек `client_settings.extra.status` (значения GTport: **1 сутки /
12 часов**). Для агрегации таких вагонов заполняется ключ **`id_status4`**
(`index|code_station_oper|time_op`).

### 5 — брошен
`code_oper == "92"` (операция «брошен»). Заполняется ключ агрегации **`id_status5`**
(та же схема `brosKey`).

### 6 — порожний в пути
`ПорожПризнак == "1"` и вагон **не** на станции назначения. Такой вагон при
**переходе** на статус 6 становится **донором перегруза** (§3.16): его груз/назначение
могут быть перенесены приёмнику без груза (§3.17, `applyStatus6Transition` /
`applyStatus6Donorship`, `stage2.go`).

### 8 — пропал (missing)
**Особый статус — не из `computeStatus`.** Назначается в `reconcileCandidates`
(`stage2.go`) вагонам, которые были в прошлом снимке, но отсутствуют в новой выгрузке.
Такие записи пишутся в таблицу кандидатов (`status9`) со статусом 8 и **в снимок
`dislocation` не попадают** (комментарий в коде: «Статус-8 в снимок НЕ идёт»). Если
вагон возвращается живым (снова 9) — защищённый 8 снимается.

### 9 — кандидат в прибывшие
Гружёный вагон уже на станции назначения (`atDest`), но факт прибытия ещё не
подтверждён (`date_prib` пусто). «Почти прибыл». Живые кандидаты статуса 9 при первом
появлении пишутся и в таблицу кандидатов (S2-1). Прогноз хода для 9 не строится
(вагон уже на месте).

### 10 — прибыл
Гружёный на станции назначения с заполненным `date_prib`. Факт прибытия. Для истории
(`vagon_history`) это веха прибытия (`date_prib`). При загрузке плана подвода
плановые данные прибывших **не очищаются** (`planmatch/apply.go` бережёт статус 10).
`date_kon` для 10 берётся из `date_op_jd` (см. ниже).

### 12 — порожний в порту / выгружен
`ПорожПризнак == "1"` и вагон на станции назначения. Трактуется как выгрузка в порту.
Для истории — веха выгрузки (`date_vigr`).

## Производные поля, зависящие от статуса

Считаются там же, в Stage 1, шаг 4 (`enrich.go`):

- **`date_kon`** (`computeDateKon`): для статуса **10** — `date_op_jd` (ЖД-время
  прибытия), для всех прочих — `time_op` (время последней операции).
- **`id_status4` / `id_status5`** (`brosKey`): ключ агрегации брошенных (5) и
  простаивающих (4) — `index | code_station_oper | time_op`. Пуст, если нет индекса /
  станции / времени операции.
- **`delay`** (`computeDelay`): просрочка доставки в сутках, если норматив
  `date_dostav` в прошлом относительно «сегодня» (МСК); иначе `nil`.
- **`id_disl`** (`computeIdDisl`): ключ поезда `index | code_station_oper | oper_s |
  date_op(ДД.ММ.ГГГГ)` из непустых компонентов — используется для группировки/матча.

## Удержание статуса (sticky 4/5)

При переносе из прошлого снимка (`carryover.go`, `enrichFromActual`) статусы **4**
(простой) и **5** (брошен) **удерживаются**, пока вагон стоит на **той же** станции
операции (`code_station_oper` не изменился). Это гасит «мигание» статуса между
выгрузками, когда свежие данные РЖД временно дают другой код. Как только станция
операции меняется — статус пересчитывается заново.

---

## Влияние статуса на прогноз хода

`forecast.go` (Stage 3, S2-5) считает время до порта (`ToGo → RaschMsk / RaschJd`)
только для вагонов **в пути**:

- **считается** для статусов `< 9` (1, 2, 4, 5, 6);
- **пропускается** для статуса **0** (операционный буфер, ещё не в ходу);
- **пропускается** для статусов `≥ 9` (9, 10, 12 — вагон уже на месте).
