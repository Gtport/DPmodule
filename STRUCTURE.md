## Как устроен backend (слои)

Поток данных идёт сверху вниз, зависимости — только «вниз»:

```
handler (HTTP, gin)  →  service (бизнес-логика, RAM-кэши)  →  repository/gorm (БД)
        ↑                          ↑                                   ↑
     port (интерфейсы)      domain (чистые типы)              parser (разбор входных файлов)
```

- **domain** — чистые типы предметной области (вагон, план, справочники), без БД и HTTP.
- **port** — интерфейсы («что умеет хранилище/интеграция»), реализуются в `repository`.
- **service** — вся логика: конвейер обогащения дислокации, приём ЛК, матч плана; держит
  состояние в RAM-кэшах (БД читается на старте, пишется атомарной подменой снимка).
- **repository/gorm** — доступ к PostgreSQL (ORM-модель отделена от доменной).
- **parser** — разбор входных файлов (Excel ЛК, JSON АСУ, Excel план) в доменные типы.
- **handler** — тонкий HTTP-слой (gin), маппинг ошибок в коды; провод в `server.go`.

Точка сборки — `cmd/server/main.go` → `internal/server/server.go` (DI, монтаж маршрутов).
