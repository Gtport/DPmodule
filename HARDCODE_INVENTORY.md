# GTport — Инвентаризация хардкода (порт-специфика)

> Каталог всех мест, где «три порта» (`at`/`ut`/`gut` → `АЭ`/`УТ-1`/`ГУТ-2`) и
> связанная с ними бизнес-логика зашиты в код, схему БД или фронт. Для каждого
> пункта: где находится, что зашито, чем заменяется в универсальной модели.
> Это план работ по выносу констант в БД. Парная пара к `ARCHITECTURE.md`;
> целевые сущности проектируются в `TARGET.md`.

**Условные обозначения сложности:** 🟢 простое (словарь/маппинг) ·
🟡 среднее (логика + схема) · 🔴 сложное (парсеры/формат/много мест).

---

## 0. Что УЖЕ data-driven (опора для рефакторинга)

Прежде чем перечислять хардкод — что уже хранится в БД и может стать основой:

- Таблица **`ports`** + модель `models.Ports` (id, name, code, okpo, location,
  проценты по родам груза, фронт-параметры). Справочник портов уже существует.
- Таблица **`max_chats`** (`name`, `chat_id`, `description`, `is_active`) +
  `MaxChatRepository`. Сами чаты — в БД; зашита только привязка порт→чат (см. 4.1).
- Справочники **`stations`**, **`marka`**, **`cargo_operations`** грузятся в RAM
  (`DirectoryCache`). Обогащение дислокации уже идёт через данные, а не код.

Вывод: инфраструктура «справочник в БД → кэш в RAM → использование» уже есть.
Универсализация = распространить этот паттерн на то, что сейчас в `switch`/`map`.

## 1. Доменная модель: тип порта

### 1.1. 🟡 Перечисление `PortType` — корень всего
- **Файл:** `cargowork.go` (models)
- **Что зашито:**
  ```go
  type PortType string
  const ( PortAT PortType = "at"; PortUT = "ut"; PortGUT = "gut" )
  ```
- **Используется в:** десятках `switch port {...}` по всему backend.
- **Замена:** код порта — строковый идентификатор из таблицы `ports.code`,
  без констант в коде. `switch` → поиск в справочнике/конфиге порта
  (`PortRegistry`).

### 1.2. 🟡 Тройное соответствие code ↔ naznach ↔ display
По системе разбросаны три «координаты» одного порта, которые нужно свести в
одну запись справочника:
- `code`: `at` / `ut` / `gut`
- `naznach` (назначение в данных дислокации): `АЭ` / `УТ-1` / `ГУТ-2`
- `display`/`shortName`: `Аттис` (для `АЭ`!), `УТ-1`, `ГУТ-2`
- **Замена:** поля одной строки `ports`: `code`, `naznach`, `display_name`,
  `short_name`.

## 2. Backend: `switch`-логика по порту

### 2.1. 🟡 SMS-план — назначение и обработка по порту
- **Файл:** `sms_plan_service.go`
- **Функции / что зашито:**
  - `getNaznachByPort`: `PortAT→"АЭ"`, `PortUT→"УТ-1"`, `PortGUT→"ГУТ-2"`.
  - `doesSubgroupMatchPort`: строгое соответствие `Naznach == "АЭ"/"УТ-1"/"ГУТ-2"`.
  - `fillCargoWorkData` → `processStandardPortData` (АТ/УТ) vs `processGUTData` (ГУТ).
- **Замена:** `naznach` берётся из справочника порта. Различие «стандартный порт
  vs ГУТ» — это различие в **профиле обработки** (ГУТ разбивает груз на
  УГОЛЬ/МЕТАЛЛ/ЧУГУН): вынести в признак порта `cargo_split_profile`.

### 2.2. 🟡 Грузовая работа — план и история по порту
- **Файл:** `cargowork_repository.go`
- **Функции / что зашито:**
  - `getPribFromHistory`: `port → naznach` (тот же маппинг) + для ГУТ возвращает
    разделённые prib/mPrib/chPrib, для АТ/УТ — только общий.
  - `getOstStFromPlan`: `port → planType`: `AT→"ma"`, `UT→"nk"`, `GUT→"ma"`.
- **Замена:** `naznach` и тип плана (`ma`/`nk`) — атрибуты порта в справочнике
  (`ports.plan_type`). Разделение по родам груза — `cargo_split_profile` (см. 2.1).

### 2.3. 🟡 Статистика прибытия — разбивка по роду груза
- **Файл:** `info_history_repository.go`, `GetPribStats`
- **Что зашито:** `switch naznach`: для `АЭ`/`УТ-1` — общий счёт; для `ГУТ-2` —
  разбивка по `cargo_group`: `УГОЛЬ` / `МЕТАЛЛ` / `ЧУГУН`.
- **Замена:** профиль разбивки груза в справочнике порта (список cargo_group →
  выходные поля), вместо зашитого `switch`.

### 2.4. 🟡 Кэш SMS-плана — порты прямо в СХЕМЕ БД
- **Файлы:** `sms_plan_cache_repo.go`, таблица `sms_plan_cache`
- **Что зашито:** физические колонки `port_at`, `port_ut`, `port_gut`; `switch
  port` выбирает, в какую колонку писать; UPSERT строит имя колонки из порта.
- **Замена:** строковая модель кэша — `(calculation_date, port_code, data, ...)`,
  одна строка на порт. Снимает «ровно три порта» из самой схемы. **Требует
  миграции БД.**

## 3. Backend: клиентский дашборд и портовый отчёт

### 3.1. 🟡 Клиентский дашборд — режимы и фильтры
- **Файл:** `client_dashboard_handler.go`
- **Что зашито:**
  - `loadBros`: `gruzpolKey := {"АЭ":"at","УТ-1":"ut","ГУТ-2":"gut"}` и SQL
    `... gruzpol_s IN ('АЭ','УТ-1','ГУТ-2')`; результирующая map с ключами
    `"at"/"ut"/"gut"`.
  - `GetGtPortSnapshot`: `switch port` → `pageMode`: `at→"АЭ+ГУТ"`,
    `gut→"АЭ+ГУТ"`, `ut→"УТ-1"` + `ganttPort`.
  - `filterTrainsByPort`: `naznachMap := {"АЭ":{"АЭ","at"}, "ГУТ-2":{...}, ...}`.
- **Замена:** список портов и их naznach — из справочника; режимы агрегации
  («АЭ+ГУТ» как совмещённое представление двух портов) — отдельная сущность
  «представление/группа портов» (`port_view`), а не зашитый `switch`.

### 3.2. 🔴 Портовый отчёт НМТП — маппинг станций по колонкам
- **Файлы:** `port_report_nmtp_handler.go`
- **Функции / что зашито:**
  - `getPortDisplayName`: `{"АЭ":"Аттис","ГУТ-2":"ГУТ-2","УТ-1":"УТ-1"}`.
  - `getNmtpSprav1Mapping`: **большой** маппинг `sprav_1 → {группа, колонка,
    имя станции, комментарий}`, РАЗНЫЙ для `ГУТ-2` (металлургия: НОВОКУЗНЕЦК,
    СМЫЧКА, РУК, КЛЦ МАРИС, ЭЛЬГА) и для `АЭ`/`УТ-1` (уголь: СУЭК, ЭЛЬГА — с
    конкретными станциями Чегдомын, Черногорские Копи, Челутай, Лен-Куз, УЛАК…).
  - `getNmtpColumnGroups`: группировка этого маппинга по колонкам.
- **Замена:** это **конфигурация раскладки отчёта** — таблица вида
  `report_column_mapping(port_id, report_type, sprav_id, group, column, name,
  comment)` (без `enterprise_id` — модель одноклиентская, см. `TARGET.md` 3.6).
  Один из самых объёмных кусков хардкода; чисто справочный по природе, переносится
  в БД относительно прямолинейно, но записей много.

## 4. Frontend: порт-специфика на клиенте

> Привязки и словари зашиты в нескольких компонентах. Сами чаты MAX — в БД
> (`max_chats`), но соответствие «порт → имя чата» дублируется в коде.

### 4.1. 🟢 `MAX_CHATS_CONFIG` — порт → чат(ы) MAX (дублируется!)
- **Файлы:** `SmsOper.tsx`, `SmsPlan.tsx`, `LoadingReport.tsx`,
  `PortReportTable.jsx`, `PortReportNmtp.jsx`
- **Что зашито (варьируется по файлам):**
  ```js
  { 'АЭ': {chat:'at', operChat:'at_o'}, 'ГУТ-2': {chat:'gut', operChat:'gut_o'},
    'УТ-1': {chat:'ut', operChat:'ut'} }          // SmsOper
  { 'АЭ':['at'], 'ГУТ-2':['gut'], 'УТ-1':['ut'], 'MARIS':['oper'] }  // PortReportTable
  ```
  Плюс `MAX_CHATS_CONFIG.all` и спец-ключ `MARIS` (клиент → оперативный чат).
- **Замена:** API-эндпоинт, отдающий привязку порт→чаты из БД (расширить
  `max_chats` связью с портом, либо таблица `port_chat_binding`). Фронт берёт
  конфиг с сервера, перестаёт дублировать.

### 4.2. 🟢 Косметика порта — цвета, имена
- **Файлы:** `SmsOper.tsx`, `PortReportTable.jsx`, `PortReportNmtp.jsx`
- **Что зашито:**
  - `portColors`: `{'АЭ':'#FFFFD9','ГУТ-2':'#CAF6FB','УТ-1':'#FFEFF4'}`.
  - `portFullNames`: `{'АЭ':'АТТИС','ГУТ-2':'ГУТ-2','УТ-1':'УТ-1'}`.
  - `getAlternatePortName`: `{'АЭ':'НМТП','ГУТ-2':'Аттис','УТ-1':'НМТП'}`.
  - `portsConfig`: массив с `enabled/apiCode/naznach/shortName/maxChat/...`.
- **Замена:** все эти атрибуты — поля справочника `ports` (`color`, `display_name`,
  `alternate_name`), отдаются API. В `ports` уже есть поле `color`.

### 4.3. 🔴 `sprav1Mapping` / `sprav1MappingGUT2` — дубль маппинга станций на фронте
- **Файл:** `PortReportNmtp.jsx`
- **Что зашито:** те же станционные раскладки, что в backend (см. 3.2), —
  отдельно для угля (СУЭК/ЭЛЬГА) и металла (ГУТ-2). **Дублирование** серверного
  маппинга на клиенте.
- **Замена:** единый источник — API из таблицы `report_column_mapping` (3.2).
  Устранить дублирование backend/frontend.

### 4.4. 🟢 Базовый URL API
- **Файл:** `config.ts`
- **Что зашито:** `API_URL = process.env.REACT_APP_API_URL || 'https://localhost:8080'`.
- **Замена:** не порт-специфика, но относится к захардкоженным дефолтам —
  вынести в env прототипа (и обратить внимание на `https` в дефолте, см. вопрос
  про HTTPS в dev).

## 5. Парсеры входных данных (форматы)

### 5.1. 🔴 Два семейства парсеров плана: МА и NK
- **Файлы:** `plan_ma_parser_old/new.go`, `plan_nk_parser_old/new.go`,
  `plan_nk_synonyms.go`, `plan_ma_service.go`, `plan_nk_service.go`
- **Что зашито:** различие форматов входных файлов планов подвода (МА vs NK),
  «старый» и «новый» варианты форм, синонимы станций (NK), привязка порта к типу
  плана (см. 2.2). Эвристики поиска заголовков, номера колонок, служебные строки
  («Остаток на 18:00», «План на DD-MM-YYYY»).
- **Замена (целевая, самая трудоёмкая):** профиль парсера как данные —
  `parser_profile(format, header_markers, column_map, synonyms, service_rows)`
  (без `enterprise_id`; детали — `TARGET.md` 3.8). Парсер становится
  конфигурируемым движком вместо набора форматных веток. **Это «этап 3» дорожной
  карты — оставить на потом.**

### 5.2. 🔴 LK-парсер: маппинг заголовков
- **Файл:** `parse_lk.go`, `getColumnIndexes`
- **Что зашито:** словари `exactMatches`/`partialMatches` (рус. заголовок →
  внутреннее поле, напр. «Номер вагона»→`vagon_c`), поиск строки заголовка по
  «Номер вагона», `generateDeterministicID` = `вагон/станция/дата`.
- **Замена:** маппинг заголовков → профиль парсера (как 5.1). Заметка: этот
  маппинг **не** порт-специфичен, но формат-специфичен — для нового предприятия
  с другой формой ЛК потребуется новый профиль.

### 5.3. 🔴 Константы скорости выгрузки
- **Файл:** `plan_nk_service.go`
- **Что зашито:** `SpeedNMTP_Nk = 18.0`, `SpeedUgTerm_Nk = 10.0`.
- **Замена:** атрибуты порта/назначения в справочнике (`ports.unload_speed`).

### 5.4. 🟢 Коды портов в имени файла
- **Файл:** `filename_formatter.go`, `getPortName`
- **Что зашито:** `AT→Аттис`, `NMTP→НМТП`, `TA→Терминал`, `PZ→Примзавод`
  (обратите внимание: коды в именах файлов отличаются от `PortType`!).
- **Замена:** маппинг код-в-имени-файла → порт из справочника
  (`ports.file_code`).

## 6. Прочие зашитые места

### 6.1. 🟡 SQL с явным перечислением назначений
- **Где:** `client_dashboard_handler.go` (`loadBros`), bros-обработчики и др. —
  `WHERE gruzpol_s IN ('АЭ','УТ-1','ГУТ-2')`.
- **Замена:** список назначений строить из справочника портов (динамический
  `IN (...)` или join на `ports`).

### 6.2. 🟡 Различие cargo-split (уголь vs металл) как «ГУТ-особенность»
- **Где:** `cargowork_repository.go`, `cargowork_service.go`, `sms_plan_service.go`,
  модель `cargowork.go` (поля `*Coal*`, `*Metal*`, `*Chugun*` JSON).
- **Что зашито:** предположение, что разделение груза на уголь/металл/чугун —
  это «про ГУТ». Поля структуры жёстко названы под три рода.
- **Замена:** обобщённый профиль рода груза порта (список cargo_group и их
  выходных полей), вместо именованных под конкретные рода полей. 🔴 по факту,
  т.к. затрагивает и модель, и репозиторий, и сервис.
- ⚠️ **След в новом коде DPmodule:** `internal/service/stage4_apply.go`,
  `cargoRod()` — switch `УГОЛЬ→coal / МЕТАЛЛ→metal / *→other` для выбора
  перерабатывающей способности `pc_*`. Устранять вместе с этим пунктом:
  колонка «род» в словаре `cargo` (миграция 000027) вместо switch.

4. **🔴 Раскладка отчётов (3.2, 4.3):** перенести станционные маппинги в

6. **🔴 Парсеры (5.1, 5.2):** профили парсеров — в самом конце, когда модель
