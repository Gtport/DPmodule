# Инструкция проекта: GTport Universal (реорганизация)

## О проекте
Реорганизация GTport — рабочего инструмента диспетчера логистического центра.
Цель — из приложения, жёстко заточенного под 3 конкретных порта
(`at`/`ut`/`gut` → `АЭ`/`УТ-1`/`ГУТ-2`), сделать универсальный прототип,
настраиваемый под любое предприятие: хардкод-константы выносятся в БД, структуры
и парсеры входных данных расширяются.

**Способ реализации (зафиксировано): пишем заново на шаблоне `TMPL_backend`
и переносим бизнес-логику из GTport.** Это greenfield, не доработка существующего
кода по месту. Старый GTport (`gtlogic`) — источник **бизнес-логики и правил**;
`TMPL_backend` — источник **архитектуры**.

Стек нового кода (по `TMPL_backend`): Go 1.26, `gin`, доступ к данным —
**`GORM` (`gorm.io/gorm` + `gorm.io/driver/postgres`, под капотом `pgx`) в
гибридном режиме** (зафиксировано): билдер GORM для CRUD и динамических
`UPDATE`; сырой SQL дословно через `gorm.Raw().Scan()` / `gorm.Exec()` для
аналитики (CTE, `DISTINCT ON`, оконные), DDL и атомарной подмены таблиц. Excel —
`excelize`, аутентификация через **Keycloak** (JWKS), логирование `zap`, конфиг
YAML + секреты в env, миграции `golang-migrate` (схему ведёт он, **не**
`AutoMigrate`), метрики Prometheus, Swagger. **Redis и MinIO не используются**
(удалены из шаблона): кэш — это RAM-движок дислокации (`actualMap`),
`DirectoryCache` и `sms_plan_cache` в Postgres; файлы — как в GTport (решение по
хранилищу файлов — отдельно). Приёмы перевода на GORM (что на билдере, что
остаётся сырым SQL) — `DPmodule/_reference/GORM_MIGRATION_GUIDE.md` (референс приёмов,
не план действий: он писался под рефакторинг старого gtlogic по месту, а мы
пишем заново).

**Время — единая шкала АСУ, без таймзон (зафиксировано).** Все даты/время берём
и отдаём ровно так, как пришли из АСУ, без перевода зон и без суффикса `Z`. Тип —
`LocalTime` (обёртка над `time.Time`: JSON и БД без `Z`, формат
`2006-01-02T15:04:05`; колонки — `timestamp without time zone`). `LocalTime`
применяется **везде** в доменных моделях, включая `VagonHistory`/`VagonOperation`
(прежний `*time.Time` в них переводится на `LocalTime` при переносе — чтобы не было
разнобоя между дислокацией и историей).

Владелец — логист, не программист; код пишется с помощью ИИ. Главный актив
проекта — бизнес-логика. Поэтому приоритет у решений, которые защищают
бизнес-логику от случайной поломки и делают код надёжно редактируемым ИИ.

## Модель развёртывания и мультиарендность (зафиксировано)
Silo: **один клиент = один экземпляр приложения = одна БД = один поток входных
данных.** Нескольких клиентов в одном процессе НЕТ; `enterprise_id` в схеме НЕ
вводим (если документ ниже где-то его упоминает — это устаревшее, см. `TARGET.md`).
Роль «клиента» играет настроечная таблица `client_settings` (синглтон).

У клиента может быть несколько портов, но **порт — это не tenant**: порт —
производный атрибут вагона (`naznach`), фильтр единого `actualMap` по вторичным
индексам, а не раздел хранилища. Поэтому «несколько портов» не меняет модель
хранения. Универсализация = сделать измерение «порт» (число и атрибуты)
конфигурируемым через БД + `PortRegistry`, а не зашитым в `switch`. Новый клиент =
новое развёртывание (своя БД, свой Keycloak realm, свой конфиг, seed
`client_settings` + `ports`). Подробнее — `TARGET.md` раздел 0.

RAM-движок дислокации (`actualMap` + индексы + конвейер обогащения) переносится
**как есть** — на нём построена вся бизнес-логика. Он одноклиентский по природе,
что и делает модель Silo единственно совместимой с решением «движок не трогаем».

## Объём: что упраздняем (зафиксировано)
Упраздняем работу с **историческими таблицами-снимками и бэкапами** старого
GTport: ротацию `disl_actual` в схему `history`, генерацию timestamped-таблиц,
ручные бэкапы и восстановление из них (и завязанные на них эндпоинты). В
greenfield-коде этот функционал просто **не пишем**.

⚠️ **Это НЕ касается бизнес-таблиц `vagon_history` и `vagon_operation`** (журнал
рейсов и история продвижения по запросу 601) — они остаются и являются частью
бизнес-логики. Слово «история» здесь перегружено: «исторические таблицы/бэкапы»
(снимки `disl_actual`, упраздняем) — это совсем другое, чем «история рейса»
(`vagon_history`/`vagon_operation`, сохраняем). При любой работе с «историей»
сверяться, о чём речь, чтобы не удалить нужное.

Атомарная подмена снимка дислокации сохраняется по **варианту B**: заливка в
staging-таблицу `disl_new` → быстрый `rename` в `disl_actual`, **без** ведения
истории снимков (см. `GORM_MIGRATION_GUIDE.md`, разд. 4 и 7.4).

## Как устроена работа в этом проекте
- **База знаний** хранит только `.md`-документы (карта системы и план). Это
  стабильный контекст, источник истины для каждого чата.
- **Код на переработку приходит в чат** — целыми файлами, не кусками. Один чат —
  одна логическая задача (одна подсистема, один-два связанных файла).
- **После переработки** соответствующий `.md` обновляется и кладётся обратно в
  базу знаний, чтобы карта оставалась актуальной.
- Считай, что без выложенного в чат кода ты его не видишь — опирайся на
  документы базы знаний и на то, что прислано в чат.

## Документы базы знаний
- `ARCHITECTURE.md` — карта GTport «как есть» (источник правды по **бизнес-логике**,
  которую переносим; **не** образец архитектуры нового кода).
- `HARDCODE_INVENTORY.md` — каталог порт-специфичного хардкода и чем он заменяется.
- `TARGET.md` — целевая универсальная модель (схемы таблиц, реестр, миграция).
- `OPTIMIZATION.md` — план улучшений GTport. Частично **уже реализован** в
  `TMPL_backend` (golang-migrate, zap, pgx, лимиты пула, единый паттерн, метрики);
  актуальны прежде всего бизнес-правила для golden-тестов (P0-1) и список
  «что не трогать».
- **Код приложения (DPmodule, по шаблону `TMPL_backend`)** — источник правды по
  **архитектуре** нового кода (слои, DI, структура каталогов).

Сверяйся с `TARGET.md` перед предложениями по архитектуре данных; с
`HARDCODE_INVENTORY.md` — при работе с порт-специфичными местами; с кодом
`TMPL_backend` — по структуре слоёв и DI. При конфликте `.md`-документов
приоритет: `PROJECT_INSTRUCTIONS.md` → `TARGET.md` → остальные.

## Правила работы с кодом
- **Сохранять поведение GTport.** Прод GTport работает и остаётся эталоном
  поведения. Новый код на `TMPL_backend` должен давать те же результаты на тех же
  входных данных. Перенос — по подсистемам, маленькими проверяемыми кусками.
- **Сначала тест, потом перенос.** Бизнес-правило фиксируется тестом до/во время
  переноса. Страховка — **golden-тесты**: зафиксировать выходы GTport на наборе
  входных фикстур и прогонять их против нового кода («новый код == поведение
  GTport»). Это замена прежнего теста эквивалентности «switch == registry»
  (раньше внутри одного кода; теперь между GTport и новой реализацией).
- **Никакого нового хардкода.** Новые порт-специфичные данные — через таблицы
  и конфиг (реестр `PortRegistry`), не через `switch`/`map` в коде.
- **Падать громко** на операциях с целостностью данных — не «log и продолжить».
- **Миграции** — идемпотентные, только добавляющие, безопасные для повторного
  запуска; применять в форке, не на проде.
- **Не переписывать модуль целиком** без явной просьбы. Минимальное изменение,
  решающее задачу.

## Эталонные паттерны (следовать в новом коде)
- **Архитектура и DI — канон `TMPL_backend`: Ports & Adapters.** Слои:
  `domain` (чистые сущности + DTO + sentinel-ошибки) → `port` (интерфейсы) →
  `service` (логика поверх портов) → адаптеры (`repository/*`) → `handler` (gin).
  Зависимости собираются **вручную в `main.go`/`server.Build`**. Прежний эталон DI
  «как `sms_plan` (factory/container)» **отменён** — это паттерн старого GTport.
- **Доступ к данным — GORM (гибрид), канон тимлида.** Билдер GORM для CRUD и
  динамических `UPDATE` (`Model().Updates(map)` — обновлять только непустые поля);
  сырой SQL дословно через `gorm.Raw().Scan()` / `gorm.Exec()` для аналитики (CTE,
  `DISTINCT ON`, оконные), DDL и атомарной подмены таблиц. ORM-модель отделена от
  доменной сущности (ORM-структура в `repository/gorm`, доменная — в `domain`).
  Правила: GENERATED-колонки помечать
  `gorm:"->"` (read-only, не уходят в INSERT/UPDATE); снапшот-таблицам (`disl_actual`/
  `disl_new`) **не** давать `gorm.Model` (никакого soft-delete `deleted_at`) и **не**
  вешать хуки на bulk (`CreateInBatches`); `SkipDefaultTransaction: true`. Схему ведёт
  `golang-migrate`, не `AutoMigrate`.
- Готового примера-сущности в репозитории нет (плейсхолдер `Item` из шаблона
  удалён). Эталон — сама структура слоёв (`domain` → `port` → `service` →
  `repository/gorm` → `handler`) и маппинг sentinel-ошибок (`domain/errors.go`)
  в HTTP-коды в `handler`. Первый перенос (`Dislocation`) задаёт образец для
  остальных сущностей. Подробные приёмы GORM и swap «вариант B» —
  `DPmodule/_reference/GORM_MIGRATION_GUIDE.md`.
- RAM-движок дислокации — особый случай: stateful-синглтон доменного слоя (не
  репозиторий), создаётся один раз в `main.go`, зависит от порта персистентности.
- Доступ к атрибутам порта — через `PortRegistry`, не через `switch port`.
- Новый фронт — `.tsx` (не `.jsx`); UI-библиотеку выбрать одну и держаться её.
