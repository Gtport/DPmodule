# GTport — Оптимизация (архитектура, надёжность, подход)

> Набор заданий по улучшению кодовой базы, отсортированный по отдаче. Документ
> рассчитан на модель разработки «владелец-логист ставит задачу, ИИ пишет код»:
> для каждого пункта дано **что менять**, **где в коде**, **почему важно** и
> **критерий „готово"** — чтобы пункт можно было прямо отдать ИИ как задание и
> проверить результат.
>
> Контекст: GTport, Go + PostgreSQL + React, диспетчерский инструмент для портов.
> Главный актив проекта — бизнес-логика; код собран с помощью ИИ. Поэтому
> приоритет у того, что **защищает бизнес-логику от случайной поломки** и делает
> код **надёжно редактируемым ИИ**. Все работы — в форке-прототипе поверх
> работающего прода; бар для изменений в работающем коде высокий.

**Шкала приоритета:** P0 — делать в первую очередь · P1 — следом · P2 — когда дойдут руки.

---

## P0-1. Тесты на бизнес-логику (главный рычаг)

**Что.** Покрыть тестами ключевые логистические правила — не весь код, а самые
дорогие в случае ошибки расчёты. Тест = ваше правило, записанное как проверка.

**Где (с чего начать, по убыванию цены ошибки):**
- Разбивка груза уголь/металл/чугун — `info_history_repository.go` (`GetPribStats`),
  `cargowork_repository.go`, `sms_plan_service.go` (`processGUTData`/`processStandardPortData`).
- Расчёт остатков и грузовой работы — `cargowork_repository.go`
  (`getOstStFromPlan`, `getPribFromHistory`).
- Фильтрация вагонов по порту/назначению — `client_dashboard_handler.go`
  (`filterTrainsByPort`), `sms_plan_service.go` (`doesSubgroupMatchPort`).
- Защита от потери данных — `dislocation_processing.go` (порог 30%).

**Почему.** Сейчас единственная проверка изменений — «вроде работает».
Логистические правила живут только в коде и в вашей голове; правка ИИ может
тихо их сломать, а узнаете вы от диспетчера через неделю. Тест ловит поломку
**до** прода и превращает вашу экспертизу в страховочную сетку.

**Как ставить задачу ИИ.** Формулировкой-правилом: «напиши тест, который
проверяет: вагон с назначением ГУТ-2 и грузом МЕТАЛЛ попадает в bucket
металла; уголь — в bucket угля; неизвестный груз — в общий счёт». Правило
описываете на своём языке, код пишет ИИ.

**Готово, когда:** ключевые правила покрыты тестами, `go test ./...` зелёный,
и при намеренной поломке правила (для проверки) тест краснеет. Образец уже
есть — `port_registry_test.go` (тест эквивалентности «switch == registry»).

## P0-2. Устранить дублирующиеся этапы обработки

**Что.** Свести три почти одинаковые функции конвейера дислокации в один
параметризованный путь (этап — параметр, а не копия).

**Где.** `dislocation_processing.go`: `processNewRecordsWithSource` (stage 0),
`processFromStage3WithSource`, `processFromStage4WithSource`. Они отличаются
деталями, но повторяют общую структуру (merge → подготовка → обогащение →
атомарная замена → сохранение).

**Почему.** Это прямой риск для AI-разработки: вы просите ИИ поправить «этап
обработки», он правит одну из трёх копий, две другие тихо расходятся — и
поведение начинает зависеть от того, через какой этап пришли данные. Одна копия
вместо трёх = одно место правки.

**Как ставить задачу ИИ.** «В `dislocation_processing.go` три функции
`processNewRecordsWithSource` / `processFromStage3WithSource` /
`processFromStage4WithSource` дублируют логику. Сведи их в одну функцию с
параметром этапа, сохранив поведение каждого этапа в точности. Сначала напиши
тесты, фиксирующие текущее поведение всех трёх, потом рефактори так, чтобы
тесты остались зелёными.»

**Готово, когда:** одна функция обработки вместо трёх; тесты, снятые с текущего
поведения, проходят без изменений; вручную проверено обновление через stage
0/3/4.

## P0-3. «Падать громко» на операциях с целостностью данных

**Что.** Операции, влияющие на актуальность/целостность данных, должны падать
видимо, а не писать warning и продолжать.

**Где.** `main.go` — запуск миграций при ошибке логирует warning и идёт дальше
(приложение может стартовать на полу-применённой схеме). Просмотреть аналогичные
«log + continue» в путях обновления дислокации и сохранения в БД.

**Почему.** Для диспетчерского инструмента тихая ошибка опаснее явной: если
данные не обновились, а интерфейс показывает старые как свежие, диспетчер примет
решение по неактуальной картине. У вас уже есть эталон правильного поведения —
защита от потери 30% записей; нужно распространить эту философию и сделать её
осознанным правилом.

**Готово, когда:** критичные к данным сбои (миграции, сохранение дислокации,
атомарная замена) приводят к видимой ошибке/остановке, а не к тихому
продолжению; решение «можно ли продолжать после этой ошибки» принято осознанно
для каждого такого места.

## P1-4. Активное оповещение о сбоях

**Что.** Слать уведомление при сбойных событиях: синхронизация не прошла N
циклов, парсинг файла упал, потеря данных отменила обновление.

**Где.** Каналы уже подключены — `telegram_service.go`, `max_service.go`.
События уже отслеживаются — `file_sync` журнал, `dislocation_update_journal`,
`LastUpdateStatus`. Не хватает связки «сбой → оповещение владельцу».

**Почему.** Вы как владелец должны узнавать о проблеме от системы, а не от
диспетчера. Наблюдаемость для такого инструмента = надёжность.

**Готово, когда:** при искусственно вызванном сбое синка/парсинга/обновления
приходит сообщение в Telegram или MAX с понятным текстом и временем.

## P1-5. Один эталонный паттерн (DI и фронт)

**Что.** Зафиксировать единый способ делать типовые вещи и следовать ему в новом
коде; старое переписывать не нужно.

**Где.**
- Backend: сборка зависимостей вручную в `main.go`/`router.go` против контейнера
  в модуле `sms_plan` (`factory.go`/`container.go`). Эталон — паттерн `sms_plan`.
- Frontend: смешаны antd и MUI, файлы `.jsx` и `.tsx`. Выбрать по одному.

**Почему.** ИИ работает надёжнее, когда в коде один образец, а не три: иначе он
видит разные примеры и выбирает случайный. Единообразие снижает «правки мимо».

**Как закрепить.** Записать выбранные паттерны в инструкцию проекта (правило
вида «новые модули собираются как `sms_plan` через factory/container; новый
фронт — только `.tsx` + MUI»). Дальше единообразие нарастает по мере изменений.

**Готово, когда:** правила записаны в инструкции проекта; новые модули/компоненты
следуют им; есть один пример нового кода, сделанного по эталону.

## P1-6. Декомпозиция бог-сервиса `DislocationService`

**Что.** Разнести ответственность: обработка/обогащение, очереди (бросы,
статус 10/5), история, экспорт, управление памятью — в отдельные компоненты с
понятными границами.

**Где.** `dislocation*.go`, `enrich_stage*.go`, `dislocation_processing.go` —
один сервис делает слишком много.

**Почему.** Большой файл/сервис переполняет контекст ИИ — он теряет картину и
правит «мимо». Маленькие модули с одной ответственностью = надёжнее каждая
сессия с ИИ. Связано с P0-2 (дубли этапов — частный случай разросшейся логики).

**Осторожно.** Это самый рискованный пункт: затрагивает ядро. Делать **после**
P0-1 (тесты), маленькими шагами, каждый — за зелёными тестами. Не одним заходом.

**Готово, когда:** выделен хотя бы один самостоятельный компонент (например,
очереди или экспорт) с тестами; поведение ядра не изменилось.

## P2-7. Структурное логирование (`log/slog`)

**Что.** Заменить `log.Printf` на структурный `slog` (в stdlib) с уровнями и
полями; не писать секреты/PII в логи.

**Где.** Повсеместно (`log.Printf` по всему backend).

**Почему.** Структурные логи проще искать при инцидентах, уровни отделяют шум от
важного — наблюдаемость для диспетчерского инструмента это надёжность. Не срочно.

**Готово, когда:** новый код пишет через `slog`; критичные пути (обновление
дислокации, синк, ошибки) логируются структурно с уровнем и контекстом.

## P2-8. «AI-читаемость» как постоянная практика

**Что.** Держать код таким, чтобы ИИ работал по нему надёжно: модули небольшие
и с одной ответственностью, имена говорящие, файлы не разрастаются на тысячи
строк, документация актуальна.

**Где.** Сквозная практика, не точечная задача. Документы
`ARCHITECTURE.md` / `HARDCODE_INVENTORY.md` / `TARGET.md` — уже вклад в это.

**Почему.** Для вашей модели разработки это часть архитектуры, а не метафора:
чем последовательнее структура, тем меньше шанс, что ИИ потеряет контекст.

**Готово, когда:** правило записано в инструкции проекта; документы
поддерживаются в актуальном состоянии при изменениях архитектуры.

## Связанные технические пункты (из обсуждения библиотек)

Не архитектура, но относятся к надёжности/безопасности; детали — в обсуждении.
Порядок для форка:

1. **`lib/pq` → `pgx`** (через `pgx/v5/stdlib`, `sqlx` остаётся) + лимиты пула
   (`SetMaxOpenConns`/`SetMaxIdleConns`/`SetConnMaxLifetime`). Драйвер `lib/pq`
   в режиме поддержки; `pgx` — актуальный. Замена — почти одна строка в `db`.
2. **Инструмент миграций** (`golang-migrate`/`goose`) вместо самописного
   `RunMigrations` — версионирование, откат, детект «грязного» состояния
   (связано с P0-3: миграция не должна тихо падать).
3. **Allowlist для динамического SQL** — там, где в запрос подставляются имена
   колонок/таблиц строкой (`sms_plan_cache_repo.go`, имена history-таблиц).
   Плейсхолдеры `$1` имена идентификаторов не закрывают — нужен белый список.
4. **`govulncheck` в CI** — сканер известных уязвимостей зависимостей.
5. **JWT из `localStorage` → httpOnly-cookie** (+ CSRF) — снижает риск кражи
   токена при XSS. Заметная переделка авторизации (`auth.go` + фронт).

---

## Чего НЕ трогать

Чтобы не тратить силы: `chi`, `golang-jwt/jwt/v5`, `excelize/v2` — современны,
менять не на что. Rate-limiter `ulule/limiter` in-memory нормален для одного
экземпляра. И сильные стороны, которые уже есть и которые трогать не нужно:
защита от потери данных (порог 30%), проверки контекста перед долгими БД-
операциями, атомарная замена таблиц, система бэкапов.

## Сводный порядок для прототипа

**P0:** тесты на бизнес-логику → устранение дублей этапов → «громкие» ошибки.
**P1:** оповещения о сбоях → эталонные паттерны → аккуратная декомпозиция ядра.
**P2:** slog → AI-читаемость как практика.
**Техдолг параллельно:** pgx + лимиты пула → инструмент миграций → allowlist SQL
→ govulncheck → httpOnly-cookie.

Всё это — задачи, которые ставятся ИИ на языке правил и структуры. Главное
правило сквозь весь список: сначала тест, фиксирующий текущее поведение, потом
изменение — особенно в ядре дислокации.
