# DPport — план работ (для руководства)

_Составлено 2026-07-05. Оценки снизу-вверх, единицы — **спринты по 2 недели**.
Ресурс: 1 разработчик (основной темп) + техлид на ревью. Границы: разделы
**«Склад»** и **«Судовые партии»** в этот план НЕ входят (перенос из gtport
отдельной волной позже)._

---

## 1. Что это и где мы сейчас

DPport — переписанный «с нуля» диспетчер дислокации вагонов для морских портов
(замена legacy-GTport). Ядро — движок обработки дислокации: приём выгрузки,
обогащение из справочников, расчёт статусов, сравнение с предыдущим снимком,
ведение истории вагона.

**Готово и влито в основную ветку (≈45% ядра backend):**

| Блок | Что сделано |
|---|---|
| Инфраструктура | Каркас сервиса (Go/Gin, слоистая архитектура), БД PostgreSQL, миграции, CI-тесты |
| Приём данных (ЛК) | Загрузка выгрузки из личного кабинета → контроль → запись в снимок |
| **Stage 1** (построчное обогащение) | Станции, идентификация порта + фильтр включённых портов, грузовые операции, **дерево статусов**, производные поля (даты, задержки, ключи) |
| **Stage 2** (межснимковая фаза, 60%) | Кэш актуального снимка в RAM; таблица кандидатов-в-прибытие (статусы 8/9); перенос данных со старого снимка (carry-over); доноры перегруза (статус 6); RAM-кэши состояния |

**Ключевая архитектурная особенность:** всё сопоставление и обогащение идёт из
оперативной памяти (справочники + снимок + кэши в RAM); БД — только для прогрева
на старте и записи. Это даёт скорость и снимает нагрузку на БД в горячем цикле.

## 2. Что осталось — этапы

### Этап A. Финализация движка дислокации (backend)
Завершаем Stage 2 — последние шаги межснимковой обработки.
- **S2-3** — обогащение новых вагонов из справочника «марка» + донорство перегруза (забор груза у вагона-донора со статусом 6).
- **S2-4** — очереди и переходы статусов (постановка/снятие «брошенных», простоев).
- **S2-5** — прогноз времени прибытия (по нормативным скоростям маршрутов).

_После этого этапа движок дислокации функционально полон на исторических выгрузках._

### Этап B. Источники данных (ingest)
- **JSON-ingest** — основной поставщик дислокации: приём по HTTP (сейчас работает только ручная загрузка из ЛК — это резервный канал). Планировщик опроса, идемпотентность, при сбое источника — автоматический откат на ручной ЛК.
- ⚠️ **Внешняя зависимость:** ручка на отдающей стороне по дислокации. Код пишем и тестируем на моках/фикстурах заранее; **боевой E2E-тест возможен только после готовности этой ручки** (Этап F).

### Этап C. История вагона и памятки
- **vagon_history** — модель и запись истории; подтверждение прибытия оператором (перевод кандидата 9 → «прибыл» 10 → запись в историю).
- **Ручка истории вагона** — получение «конечной» истории по заданному паттерну (дёргаем по расписанию/паттерну). ⚠️ Внешняя зависимость: готовность отдающей ручки.
- **Памятки** — получение данных из памяток; **записываются ТОЛЬКО в vagon_history** (в текущий снимок дислокации не идут). ⚠️ Внешняя зависимость: источник памяток.

### Этап D. API для фронта (read-side)
- Хендлеры чтения: список дислокации, фильтры, карточка вагона + история, справочники.
- Админ-функции: очистка «вечных» пропавших (статус 8), редактирование настроечных таблиц (порты, скорости маршрутов).

### Этап E. Фронтенд (следует за backend)
Начинается, когда read-API стабилизирован. Экраны: каркас (авторизация, layout) →
таблица дислокации + фильтры → карточка вагона и история → подтверждение прибытия
→ админ-панели настроек.

### Этап F. Тестирование и приёмка
- Юнит/интеграционные тесты — **сквозняком по каждому слайсу** (так работаем с самого начала, отдельного «этапа тестирования в конце» не откладываем).
- **E2E с боевыми ручками** — как только отдающая сторона выдаёт эндпоинты (дислокация JSON, история вагона, памятки). Это внешние вехи, вне нашего контроля по срокам.
- **UAT** — приёмка владельцем-логистом на реальных данных.
- Стабилизация, багфикс, деплой на `app.gtport.ru` (HTTPS уже готов).

## 4. Внешние зависимости (главный риск сроков)

Сроки Этапа F напрямую зависят от готовности **отдающей стороны**. Код мы пишем
и покрываем тестами на фикстурах заранее, но боевой прогон невозможен без ручек:

1. **Ручка дислокации (JSON/HTTP)** — блокирует боевой ingest.
2. **Ручка истории вагона** (дёргаем по паттерну) — блокирует тест истории.
3. **Источник памяток** — блокирует функционал памяток.

👉 _Просьба к руководству: закрепить ответственных и сроки по этим трём ручкам —
это критический путь финальной приёмки._

## 5. Как ускорить (рычаги)

- **+1 фронтенд-разработчик** → фронт (спринты 5–6) параллелится с backend
  (спринты 3–4): экономия ≈ 3–4 недели.
- **Ранняя выдача боевых ручек** → E2E не упирается в конец, тестируем по мере
  готовности блоков.
